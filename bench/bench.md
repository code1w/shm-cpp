# Echo Benchmark: Unix Socket vs Shared Memory

## 测试环境

- Linux 3.10.0, GCC 11.2.0, `-O2`
- `fork()` + `socketpair()` 单进程内 server/client
- Ping-pong echo：client 发一条 → server 原样回送 → client 收到后再发下一条
- 每组预热 1000 轮，不计入统计

### 对比

| msg_size | Socket RTT (ns) | SHM RTT (ns) | SHM 加速比 |
|----------|-----------------|---------------|-----------|
| 64B      | 11,005          | 999           | **11.0x** |
| 256B     | 9,646           | 777           | **12.4x** |
| 1KB      | 9,155           | 923           | **9.9x**  |
| 4KB      | 9,961           | 1,926         | **5.2x**  |
| 64KB     | 31,699          | 32,852        | ~1.0x     |
| 512KB    | 127,627         | 323,799       | 0.4x      |

## 结论

- **小消息（64B ~ 4KB）**：共享内存 RTT 比 Unix socket 低 **5 ~ 12 倍**。socket 每次 `write`/`read` 需要两次用户态-内核态切换和内核 buffer 拷贝；共享内存直接在用户态 `memcpy` 到 mmap 区域，无系统调用开销。
- **中等消息（64KB）**：两者性能接近。此时数据拷贝本身成为瓶颈，系统调用的固定开销占比降低。
- **大消息（512KB）**：socket 反而更快。内核 socket 对大块数据传输有优化（页面级 copy、管道合并），而 ring buffer 需要完整 `memcpy` 且 busy-poll 的 cache 压力更大。
- **适用场景**：共享内存方案在低延迟、小消息高频场景（金融交易、实时控制、游戏服务器内部通信）优势显著；大消息批量传输场景下 socket 仍具竞争力。
